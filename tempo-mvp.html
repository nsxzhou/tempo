设计原则

v2.0 核心方法论：假设驱动的 MVP

v1.1 的问题是：把最致命的假设（用户想要 AI 排期吗？用户愿意安装独立 App 吗？）的验证放在了"产品做完之后"，而不是"写代码之前"。v2.0 的核心转向：先验证最冒险的假设，再用代码实现已验证的需求。

致命假设一览

级别 假设 验证时机
致命 H1：用户愿意让 AI 替自己排时间块 Phase 0（Wizard-of-Oz 实验，零代码）
致命 H2：思源用户愿意安装并使用独立 App Phase 0（用户访谈）→ Phase 1.5（Dogfooding 验证）
致命 H3：LLM 排期质量达用户采纳率 >60% Phase 0（离线 Benchmark）→ Phase 2（真实数据验证）
重要 H4：火山引擎 ASR 真实环境识别准确率达标 Phase 1 启动时技术 Spike（1-2 天）
重要 H5：思源用户群体足够大支撑产品成长 Phase 0（社区调研估算）
01

为什么要做这件事

现有待办工具在"智能规划"和"笔记生态打通"两个维度上同时缺失。思源笔记用户缺少一个原生的任务管理入口，而通用待办工具又无法与个人知识管理体系联动。

问题空间

🧩
工具割裂

任务管理在滴答清单，项目笔记在思源，日程在系统日历。三个工具之间没有数据流，每天要在"我今天该做什么"上反复切换上下文。

🤖
AI 规划停留在概念

滴答清单至今没有 AI 排期功能，Todoist Assist 只做任务拆解不做时间安排。"帮我把今天的任务排进时间块"仍然是手动活。

📝
笔记与任务脱节

思源笔记中没有原生的待办管理。在文档里写的 action item 需要手动搬到外部工具，状态变更也无法回写到笔记。

🔄
复盘数据分散

任务完成情况、时间花费、习惯追踪的数据散落在不同 app 中。想做一次周复盘，得手动从三个地方搬数据。

02

目标用户

👤
核心用户画像：知识工作者 + 思源笔记用户

已经在用思源笔记做个人知识管理，习惯在笔记中记录项目计划和行动项，但苦于没有一个好用的待办工具与笔记打通。每天面对 10-30 个任务，需要清晰的优先级和时间安排。

思源笔记重度用户 独立开发者/自由职业 效率工具爱好者 知识工作者
⚠️ 市场规模约束 思源笔记用户总量估算在数万级（对比 Obsidian 百万级），其中"重度用户"（每天使用且写 action item）可能仅几千人。这是产品的自然天花板，但也是一人公司的合理规模。关键是：需要在 Phase 0 通过社区调研确认可触达用户量级，评估是否足以支撑持续开发。

MVP 功能范围（v2.0 重构）

核心变化：v1.1 的"P0"实际上是 v1.0 的完整功能清单。v2.0 遵循"先验证致命假设，再投入工程资源"的原则，将 Phase 1 收窄为验证核心假设的最小功能切片，砍掉约 30% 延后功能。

P0 — Phase 1 最小闭环（6-8 周）

📋
任务管理（基础版）
  • 任务 CRUD：标题、优先级、截止日期、列表组织
  • 智能过滤器（今天 / 本周 / 已过期 / 自定义）
  • 自然语言日期解析（本地规则引擎，不用 LLM）
  • 子任务、标签、重复规则 — 延后到 Phase 2
📅
日历视图（精简版）
  • 月视图：任务按日期标记，点击展开列表
  • 周视图：时间轴 + 任务块
  • 日视图 — 延后到 Phase 2
  • 系统日历订阅 — 延后到 Phase 2
  • 拖拽排期 — 延后到 Phase 2
🔗
思源单向同步 提前
  • 思源 Petal 插件：扫描文档中未完成任务块
  • 一键导入到 Tempo，自动写入 custom-tempo-id 关联
  • Supabase webhook 接收导入请求
  • 状态回写 — 延后到 Phase 3
  • Widget 嵌入 — 延后到 Phase 3
🧭
体验基础 新增
  • 简单 onboarding（首次打开 3 屏引导）
  • App 内反馈入口（摇一摇 / 设置页按钮）
  • 本地通知提醒（到期前 15 分钟 / 到期时）
  • 本地缓存 + 在线优先同步（简化离线方案）

关联机制

思源笔记支持给块添加 custom- 前缀的自定义属性。用 custom-tempo-id 存储任务 UUID 作为关联键。Phase 1 插件在导入时写入此属性；Phase 3 双向同步通过它定位对应块做状态回写。

AI 排期设计（Phase 2 交付，Phase 0 前置验证）

⚠️ 前置条件：AI 排期不进入 Phase 1 代码开发。Phase 0 先通过 Wizard-of-Oz 实验（见第 10 节）验证"用户真的想要 AI 排期"，Phase 1.5 根据用户反馈决定是否启动 Phase 2 开发。

排期算法流程（设计保留，Phase 2 实现）

收集待办池
→
获取日历事件
→
读取精力曲线
→
LLM 排期
→
用户确认/调整
LLM Prompt 设计要点 输入：今日待办列表（含优先级、预估时长、截止日期）+ 日历已有事件 + 用户精力曲线。
约束：高优先级任务排在高精力时段；同一时间块不超过 90 分钟；截止日期近的任务优先；留出 15 分钟缓冲。
输出：JSON 格式的时间块列表，每条包含 task_id、start_time、end_time、reason。
Phase 0：准备 50 个排期场景，让 LLM 产出结果，请 3 个目标用户打分，验证 H3。

风险与应对（v2.0 扩充）

技术实现风险（v1.1 覆盖，保持）

思源 API 版本迭代频繁 应对：锁定最低支持版本，插件内做 API 版本检测 + 降级逻辑。
LLM 排期结果不稳定 应对：Phase 0 先做 Benchmark；缓存 + 温度参数调低 + 输出 schema 校验；给用户"重新排期"和手动微调逃生通道。
双向同步冲突 应对：Phase 3 阶段用 last-write-wins + updated_at 比对，冲突时在 App 端提示用户选择。
Supabase 免费额度 应对：初期个人使用不会超；监控用量，必要时切 Docker 自部署。

🚨 v2.0 补充：市场与行为风险（v1.1 完全缺失）

🔴 市场规模风险（最被低估） "思源笔记重度用户"仅几千人，即使 100% 转化也难以支撑产品持续发展。
应对：Phase 0 先做用户规模估算；考量是否能扩展到 Obsidian 用户或其他笔记生态；评估一人公司模式下几千用户是否足够。
🔴 单人技能广度风险 项目需同时精通 Flutter + Supabase + TypeScript（思源插件）+ LLM Prompt 工程 + 火山引擎 ASR。单人同时覆盖 5 个领域，容易每个模块都做 60 分。
应对：Phase 1 砍掉 LLM 和 ASR 两个领域，专注 Flutter + Supabase + TypeScript 三个核心栈。
🟠 LLM API 成本随使用增长 100 用户每天 AI 排期 ≈ $15/月，加上 NLP 解析和拆解会更高。方案完全没有成本和定价模型。
应对：Phase 2 前做成本测算；预留免费额度上限 + Pro 订阅分层。
🟠 竞品反应风险 如果验证成功，滴答清单或 Todoist 可能在 1-2 个版本内跟进 AI 排期。
应对：思源深度集成是更难复制的护城河；优先在思源生态建立口碑壁垒。
🟡 跨平台体验一致性 日历拖拽在 iOS 和 Android 原生行为差异大，Flutter 实现可能与原生体验有差距。
应对：Phase 2 日视图拖拽前先做平台行为调研，必要时用 platform channel 调用原生组件。
🟡 火山引擎 ASR 网络依赖 + 无 Flutter SDK 流式识别需稳定网络；可能需通过 platform channel 或 WebSocket 直连。
应对：Phase 2 启动时先做 1-2 天技术 Spike 验证 WebSocket 对接可行性。

前置验证实验设计 v2.0 新增

MVP 实验设计的核心原则：在写代码之前验证最冒险的假设。以下实验应在 Phase 1 代码开发之前完成，成本为零或极低。
🧪
实验 1：Wizard-of-Oz AI 排期验证
维度详情
验证假设H1：用户愿意让 AI 替自己排时间块
方法找 5 位目标用户提供一周任务+日历 → 你手动按 AI 规则排期 → 每天发结果 → 一周后访谈
判定≥3/5 认为"比自己排明显更好" → Go；否则 → 转向建议模式
成本1 周，零代码
产出物排期结果 + 执行偏差记录 + 用户访谈纪要
🎤
实验 2：思源用户使用意愿访谈
维度详情
验证假设H2：思源用户愿意安装并使用独立 App 管理任务
方法思源社区招募 10 人，展示概念原型（线框图/PPT），问"你会安装吗？为什么？"
判定≥5/10 表示愿意 → Go；≤3/10 → 考虑纯插件+Web 端方向
成本1-2 周，需准备概念原型（Figma/PPT）
产出物用户画像验证 + 使用场景优先级 + 价格敏感度
📊
实验 3：LLM 排期质量基准测试
维度详情
验证假设H3：LLM 排期质量达用户采纳率 >60%
方法构造 50 个排期场景（含不同任务量/日历冲突/精力模式），让 LLM 产出，请 3 个目标用户打分（1-5）
判定平均分 ≥4/5 → Go；<3/5 → 简化排期逻辑
成本1 周，需 LLM API 调用费用（~$5-10）
产出物50 个排期结果 + 用户评分 + Prompt 调优记录
📈
实验 4：市场规模快速估算
维度详情
验证假设H5：思源用户群体足够大支撑产品成长
方法思源社区活跃用户数 + GitHub Star 趋势 + 社区帖子"你用什么待办工具"回复量估算
判定可触达 ≥500 人 → Go；<200 人 → 评估拓展到 Obsidian 等生态
成本半天
产出物用户规模估算表 + 触达渠道分析

假设失败标准与 Pivot 方向 v2.0 新增

好的 MVP 方案不仅知道如何验证假设，还知道假设被证伪后往哪走。以下为每个致命/重要假设定义了失败标准和 Pivot 方向。
假设失败标准Pivot 方向
H1 用户想要 AI 排期 WoZ 实验中 ≤2/5 用户认为有用 转向"智能建议但不自动排期"模式：AI 推荐时间块，用户手动确认每个任务的时间；或做成纯思源插件（不独立 App）
H2 用户愿意用独立 App Dogfooding 14 天后 <3 人留存 转向纯思源插件 + Web 端：核心交互在思源内完成，App 退化为通知和快速查看工具
H3 LLM 排期质量达标 采纳率 <40% 或 >30% 用户手动大幅调整 简化排期：减少 LLM 依赖，用规则引擎 + 用户手动排期为主，LLM 只做"今日建议"不自动排
H4 语音创建被采用 语音占比 <10% 且反馈"不需要" 降低语音优先级，聚焦文本交互体验；保留 ASR 能力在系统级快捷指令入口
H5 市场规模足够 可触达 <200 人 拓展到 Obsidian 生态（更大的用户基数）；或降低预期，定位为思源生态精品插件而非独立产品

用户获取与分发策略 v2.0 新增

v1.1 方案对"怎么做产品"考虑充分，但对"用户怎么知道这个产品"完全空白。思源社区是主要渠道但规模有限，需要多层触达策略。

触达渠道（按优先级）

渠道方法适用阶段
🥇 思源社区 在社区发"概念介绍"帖 → 收集意向用户 → 内测邀请制（限量 20 人） Phase 0 开始
🥈 少数派 撰写"效率工具困境"类文章，自然引出 Tempo 的解法 Phase 2 上线后
🥉 V2EX / 即刻 在效率工具相关讨论中分享经验，附产品链接 Phase 2 上线后
4 GitHub 开源思源插件代码，通过思源插件市场自然引流 Phase 1 起
5 用户推荐 早期用户完成 14 天使用后，请他们帮忙在社区写推荐帖 Phase 1.5 起

早期用户策略

内测邀请制（限量 20 人）
1. 管理预期：明确告知这是早期版本，功能不完整，但会认真对待每条反馈
2. 降低流失：限量感 + 创始人直接沟通 → 用户更愿意忍受早期瑕疵
3. 反馈质量：少量深度用户 > 大量浅度用户，更适合发现产品方向性问题
4. 种子传播：满意的内测用户是最可信的推荐源

运营节奏

Phase 1：邀请 5-10 人内测
→
Phase 1.5：用户访谈 + 优化
→
Phase 2：扩大内测到 20 人 + 少数派发文
→
Phase 4：公开测试
