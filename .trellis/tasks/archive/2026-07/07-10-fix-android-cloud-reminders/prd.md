# 修复 Android 云端任务提醒

## Goal

修复 Android 提醒链路：重复任务每天只发送当天 occurrence 的一条提醒；已同步云端的任务在 Tempo 退到后台、锁屏、进程被系统回收或从最近任务划掉后，仍可由服务端 FCM 在正常联网条件下 2 分钟内送达。

## Background

- 当前本地 `NotificationService` 会为重复任务预调度未来 14 次，应用启动还会全量重排，存在旧排程残留和集中展示风险。
- 当前已部署 `send-task-reminders` Edge Function 与 FCM 设备注册，但客户端没有前台消息展示、注销 token 禁用和本地/云端所有权切换。
- 服务端当前按当前分钟查找候选并使用 delivery 唯一索引，但失败记录不会在 2 分钟窗口内重试，也缺少可测试的纯候选计算边界。

## Requirements

### R1 提醒所有权

- 未登录任务使用本地提醒。
- 已登录且 `syncPending=true` 的任务使用本地临时兜底。
- 已同步云端的任务不保留本地排程，只由服务端 FCM 提醒。
- 重复任务本地兜底只排下一次有效 occurrence，不再预排未来 14 次。
- 新版启动时清理旧本地排程，再只为本地所有权任务重建。
- 云端同步成功后取消对应任务的本地排程。

### R2 FCM 客户端生命周期

- 登录态恢复/切换、token 刷新、提醒开关变化时同步设备记录。
- 注销前禁用当前设备 token。
- 前台收到 FCM 时显示一条系统通知。
- payload 包含稳定 `reminderKey`、`taskId`、`occurrenceDate`、`reminderAt`。
- 同账号所有已登录且启用设备均接收。

### R3 服务端当天 occurrence 与不补发

- 每台设备按其时区只计算当前本地日期和当前分钟窗口。
- 有时间任务按任务时间；全天任务当天 08:00。
- 重复任务应用 RRULE、COUNT/UNTIL/recurrence_end、完成和例外。
- 已错过超过 2 分钟的提醒不补发。
- 瞬时失败最多在 2 分钟内有限重试；成功与永久失败保持幂等。

### R4 用户与运维行为

- 设置文案明确“具体时间按时提醒，全天任务当天 08:00 提醒”。
- 普通后台/划掉/锁屏属于保证范围；Android 强行停止不在保证范围。
- Cron 每分钟调用必须有可执行、可验证的部署脚本或迁移说明。

## Acceptance Criteria

- AC1：每日重复任务在单个自然日每设备最多生成/展示一条对应 occurrence 提醒。
- AC2：应用启动不会为已同步任务创建本地排程；旧排程被清理。
- AC3：未同步任务只排最近一次；同步成功后该本地排程被取消。
- AC4：前台、后台、锁屏、划掉场景使用同一 payload 语义，前台仅展示一次。
- AC5：断网或 Cron 中断越过 2 分钟后恢复时不补发历史提醒。
- AC6：注销后当前 token 禁用；重新登录后按本地提醒开关恢复。
- AC7：服务端失败可在 2 分钟内重试，成功后不重复发送。
- AC8：Flutter format/analyze/test 和 Edge Function 测试通过。

## Out of Scope

- Android 系统设置中的“强行停止”后继续推送。
- 历史提醒补发、提醒提前量、短信/邮件渠道。
- iOS 真机验收。
